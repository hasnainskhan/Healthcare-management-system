GoodPeople Medical Centre — ABI Integration & Deployment Summary
===============================================================

Date: 2026-04-14
Project: Smart-Healthcare-Management-System


1) Scope / Objective
--------------------
Integrate Abi (ABI) into the system to:
- Allow users to login to our app (existing email/password auth).
- Onboard/register users into Abi via Abi REST API (server-side).
- Store Abi identifiers in our database.
- Embed the Abi GoApp widget via iframe using the returned unique identifier.
- (Optional) Accept Abi webhook callbacks securely (HMAC signature verification).
- Provide Docker-based deployment (MongoDB + Backend + Frontend via nginx).
- Apply branding updates (logo + metadata + footer credit).


2) ABI API Integration (Option 1 — confirmed by client)
-------------------------------------------------------
Reference docs:
- https://docs.abi.ai/abi-api/users/create-user

Confirmed by client:
- Base URL: https://client-api.abi.ai
- Create user: POST /user
- Unique identifier field: uniqueId
- Partner name: goodpeople
- Supported default pair: language=en, physicianCountry=ie

Auth method:
- Token exchange (required):
  - POST /partner/authorization/token with header: x-api-key: <ABI_API_KEY>
  - Then all API calls use Authorization: Bearer <access_token>

Important security rules:
- ABI_API_KEY is stored server-side only (BACKEND/.env or production .env).
- Do NOT expose ABI_API_KEY in the frontend.


3) Backend Changes
------------------
3.1 New/updated routes (JWT-protected)
- Base path: /api/abi
  - GET  /api/abi/session
    Returns onboarding status + widget_url (if onboarded) + any stored GP review/payment fields.

  - POST /api/abi/onboard
    Creates Abi user via POST /user and stores uniqueId in MongoDB.
    Uses defaults from env for partnerName/language/physicianCountry unless overridden in request body.

  - GET  /api/abi/remote
    Fetches Abi user info using /user/search?uniqueId=<id> (server-to-server).

3.2 Webhook endpoint (optional)
- POST /api/abi/webhook
  - Mounted with express.raw({ type: "application/json" }) to verify HMAC signatures.
  - Signature header supported: X-Abby-Signature / X-Abi-Signature
  - Secret: ABI_WEBHOOK_SECRET (optional; required for verification)
  - Stores incoming event details into user record where applicable.

3.3 ABI API client implementation
File: BACKEND/Services/abiApiClient.js
- Handles token exchange and caching (access_token + refresh_token).
- Uses Authorization: Bearer <access_token> for subsequent calls.
- Retries 5xx responses with exponential backoff.

3.4 Data model additions
File: BACKEND/Models/UserModel.js
Added fields for Abi mapping & status:
- abiUserId (stores Abi uniqueId)
- abiInstanceUrl (reserved; not used in current Abi Option 1)
- abiOnboardIdempotencyKey, abiOnboardedAt
- abiGpReviewSummary, abiGpReviewedAt
- abiPaymentStatus

3.5 Production server startup fix
File: BACKEND/index.js
Change: server now listens in Docker/production when executed directly:
- Previously, app.listen() ran only when NODE_ENV !== "production".
- Now, it runs when require.main === module.
This fixed nginx -> backend 502 errors in production.

6) Deployment (Server) — Docker Compose
---------------------------------------
Added:
- BACKEND/Dockerfile
- frontend/Dockerfile
- frontend/nginx.conf (serves SPA + proxies /api/ -> backend:5000)
- docker-compose.yml (root): mongo + backend + frontend
- .env.production.example (root): server configuration template

Deploy steps (high level):
1) On server, install Docker + docker compose (v2 recommended).
2) Place project on server (git clone or upload).
3) Copy .env.production.example to .env and fill:
   - JWT_SECRET (required)
   - FRONTEND_ORIGIN (e.g. http://157.173.113.20)
   - ABI_API_KEY (required for Abi calls)
4) Run:
   docker compose up -d --build

Notes:
- In production, nginx container exposes port 80.
- All API calls should go through /api/... and are proxied internally to backend.
